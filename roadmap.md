## 竞品格局总结

在 GitHub 上找到的主要同类项目有这几类：

**Android 原生工具（最成熟）**

- `PortAuthority` — 速度极快的端口扫描，多线程 TCP，支持 LAN/WAN，无服务指纹，无安全检测
- `usamaiqb/network-scanner` — 近期活跃，综合了 mDNS / SSDP / ARP / NetBIOS，支持 OS 指纹推断，但仍无安全弱点检测层
- `csicar/Ning` — 轻量设备发现，F-Droid 生态，功能单一

**桌面/CLI 工具（参考价值高）**

- `goscan` — Go 实现，SQLite 状态持久化，nmap 集成，适合做 Probe Agent 的参考设计
- 各类 nmap 封装器 — 服务枚举完善，但不适合移动端直接移植

**微信小程序现有探索（几乎空白）**

小程序侧目前只有腾讯官方的"漏洞扫描"入口（针对小程序自身后端，非通用工具），以及一些辅助渗透的外部工具（KillWxapkg 等反编译工具）。**做 LAN 安全工具的小程序几乎是空白**，这既是机会也说明有原因——平台限制是真实挑战。

## 关键技术约束（影响架构决策）

你需要在设计阶段就接受这几个硬限制：

**小程序 TCP 直连** 支持同局域网非本机 IP，但有一整张黑名单端口（22 / 3306 / 6379 / 8000–8100 等全被禁），每 5 分钟只能建 20 个 socket。这个限制决定了你**必须引入 Probe Agent 做代理**，而不能纯小程序扫描。

**mDNS 的 iOS 限制** — iOS 微信 7.0.18 以上已无法使用 mDNS 相关接口，Android 正常。所以 iOS 端只能靠 TCP 探测 + UDP SSDP 兜底。

**域名白名单** — `wx.request` 连 IP 地址在局域网场景下豁免，但公网请求必须走已备案域名。Shodan / FOFA 接口必须通过你自己的中转服务。

## Probe Agent 选型建议

Go 写单文件二进制最合适，原因：

- 无需用户安装运行时
- 跨 ARM（树莓派 / NAS / 旧 Mac）一套二进制
- 通过 WebSocket 向小程序推流扫描结果
- 同时承担高危端口代理、弱密码检测、TLS 审计这些小程序无法直接做的事情

用户首次使用时扫码安装探针（或手动下载），之后小程序自动发现局域网内的探针服务（通过 mDNS 或固定端口），体验类似 HomeKit 配对流程。

---

## 合规红线（必须提前设计）

微信审核对网络扫描类功能敏感。建议：
- Phase 3（弱口令/凭据检测）做成"仅限已授权设备"模式，用户需签署声明
- 小程序后台接口声明用途，避免被扫描系统误判
- 考虑走腾讯云企业认证白名单（安全工具类）
